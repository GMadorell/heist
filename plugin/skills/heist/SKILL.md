# /heist

You are driving the heist orchestrator in the current project.

## No arguments

Run `heist list`, filter to rows whose `stage` isn't `done`.

- Zero active rows: tell the human nothing is in progress, point them to `/heist:heist [<mode>] <description of the change>`. Stop — don't read further docs.
- One or more active rows: read `resume.md` (in this skill's directory) and follow it.

## With a description or a plan

`/heist [<mode>] [--base <branch>] [--slug <slug>] <description>` (leading tokens may appear in any order before the description).

`<mode>` is an optional token, matched case-insensitively against `heavy`, `medium`, `light`.
`--base <branch>` is an optional leading token that specifies the base branch for the worktree. When present, it's stripped and carried through step 2 of `pipeline.md`.
`--slug <slug>` is an optional leading token that pins the heist's slug instead of letting the orchestrator propose one. When present, it's stripped and carried through step 2, which uses `<slug>` verbatim. `heat.md` emits it so a stacked piece's branch name (`heist/<slug>`) is deterministic and matches what a later piece names as its `--base`.

- Mode given: strip it off; the rest is the description.
- `--base <branch>` given: strip it off; carry the branch value through step 2 of `pipeline.md`, so the `heist worktree add <slug>` call becomes `heist worktree add --base <branch> <slug>`. Stacked pieces generated by `heat.md` carry `--base heist/<earlier-piece-slug>` verbatim, never a bare branch like `main`, so the reference stays valid when pieces run long after `heat.md` was generated; the first/unstacked piece omits `--base`.
- `--base <branch>` omitted: step 2 runs `heist worktree add <slug>` exactly as before, no behavior change.
- `--slug <slug>` given: strip it off; carry it through step 2, which uses it verbatim.
- `--slug <slug>` omitted: don't propose one here; carry nothing, step 2 of `pipeline.md` proposes a slug inline.
- Mode omitted: judge scope from the description/plan (files touched, new capability vs tweak, blast radius) and ask via `AskUserQuestion`, pre-selecting the fitting mode with a one-line reason. Never default silently; human still confirms. Ambiguous/thin description -> recommend `heavy`.
  - `heavy` — full pipeline (Fence, Forger/score.md, Wheelman+Muscle, Cleaner). Multi-module, new capability, real blast radius, or open design questions.
  - `medium` — heavy minus Fence review. Moderate, well-understood changes.
  - `light` — plan + human review, then direct implementation + manual crit. Small, single-file/module, well-understood changes.

### Plan detection

After stripping the mode, `--base <branch>`, and `--slug <slug>` tokens (any that are present), check other parts of the input for file paths.
No file paths found -> normal description path, read `pipeline.md` (in this skill directory), and run it from step 1, carrying `<mode>`, `<branch>` (if given), `<slug>` (if given), and the description.
File paths found -> if any empty or unreadable, halt. Show diagnostic to user. If all file paths are good, continue to next subsection.

### Plan-based heist: confirm before spending resources

1. Canonicalize filepaths to absolute paths.
2. Ask the human via `AskUserQuestion`: list the source-set file(s) with absolute path and size, the total combined size, the mode; ask whether to proceed as a plan-based heist.
   - Declined: fall back to the normal description path — read `pipeline.md` from step 1, carrying `<mode>`, `<branch>` (if given), `<slug>` (if given), the prose (if any) and the files, which in this case are used as context. If there was no description or the files didn't have context, ask the human for a description via `AskUserQuestion` first.
   - Confirmed: read `pipeline.md` (in this skill's directory) and run it from step 1, carrying `<mode>`, `<branch>` (if given), `<slug>` (if given), "plan-based heist", the source set (absolute paths), and the prose through.
